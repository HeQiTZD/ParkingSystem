# 配置文件使用结构文档

## 1. 概述

本文档描述停车场管理系统中配置文件（`config.json`）的架构设计、使用方式和演进过程中解决的核心问题。

### 1.1 配置文件作用

`config.json` 集中管理系统的所有可配置参数，避免硬编码，使部署和调优无需重新编译：

| 配置节 | 内容 | 典型用途 |
|--------|------|----------|
| `database` | 主机/端口/库名/账号密码 | 数据库连接 |
| `parking` | 停车场名/容量/每小时费率/免费时长 | 车位管理与计费 |
| `camera` | 摄像头索引/分辨率/帧率 | 视频采集 |
| `recognition` | 模型路径/置信度阈值 | 车牌识别 |
| `system` | 应用名/版本/语言 | 全局信息 |

### 1.2 当前配置文件示例

```json
{
    "camera": { "fps": 30, "height": 1080, "index": 0, "width": 1920 },
    "database": { "host": "localhost", "name": "car", "password": "root", "port": 3306, "username": "root" },
    "parking": { "capacity": 500, "freeMinutes": 15, "name": "停车场1", "price": 12 },
    "recognition": { "confidenceThreshold": 0.7, "modelpath": "./model/" },
    "system": { "appName": "停车场管理系统", "language": "zh_CN", "version": "1.0.0" }
}
```

## 2. 设计思路

### 2.1 架构演进

配置文件管理经历了三个阶段：

```
阶段一：栈对象直读
  各业务处随时 new InitFile → loadConfig() → getXxx()
  └ 问题：重复磁盘 IO、多实例数据不同步

阶段二：参数向下传递
  MainWindow 读一次 → 通过函数参数传给 DatabaseManager / Car
  └ 问题：DatabaseManager 仍 #include "initfile.h"，耦合未彻底解除

阶段三：Meyers 单例（当前）
  全局唯一 InitFile::instance()，启动时 loadConfig() 一次性读盘
  └ getter 仅读内存 QJsonObject，零磁盘 IO
```

### 2.2 设计原则

| 原则 | 体现 |
|------|------|
| **单一数据源** | 配置全局唯一副本，避免多实例不一致 |
| **读写分离** | 启动时一次性加载；setter 显式触发保存；析构不自动写盘 |
| **最小依赖** | 数据库层（DatabaseManager）/ 值对象（Car）不直接依赖 InitFile |
| **向下传参** | UI 层读取配置，通过函数参数传递给下层 |
| **封装保护** | 构造/拷贝/移动全部 `= delete`，只允许 `instance()` 访问 |

### 2.3 分层架构

```
┌──────────────────────────────────────────────────────┐
│  UI 层（MainWindow / ConfigInitDialog）                │
│    └ 调用 InitFile::instance().getXxx() 读取配置      │
│    └ 通过参数向下传递（不传递 InitFile 引用）           │
├──────────────────────────────────────────────────────┤
│ 业务层（Car::calculateFee）                            │
│    └ 接收 hourlyRate / freeMinutes 参数               │
│    └ 不直接读取配置                                    │
├──────────────────────────────────────────────────────┤
│ 数据层（DatabaseManager）                              │
│    └ checkIn/checkOut 接收 parkingName 参数           │
│    └ deleteCarRecord 通过单例读取（特例）              │
├──────────────────────────────────────────────────────┤
│ 配置层（InitFile 单例）                                │
│    └ 启动时 loadConfig() 一次性读盘                    │
│    └ getter 仅读内存 configData                        │
│    └ setter 显式 emit + saveConfig()                   │
└──────────────────────────────────────────────────────┘
```

## 3. 关键逻辑

### 3.1 单例实现（Meyers 模式）

```cpp
// initfile.h
class InitFile : public QObject {
public:
    static InitFile &instance();   // 唯一公开入口
private:
    explicit InitFile(QObject *parent = nullptr);
    InitFile(const InitFile &) = delete;
    InitFile &operator=(const InitFile &) = delete;
    InitFile(InitFile &&) = delete;
    InitFile &operator=(InitFile &&) = delete;
};

// initfile.cpp
InitFile &InitFile::instance() {
    static InitFile instance;   // 首次调用构造，C++11 保证线程安全
    return instance;
}
```

**为什么选 Meyers 单例而非 `new` 堆分配：**
- 进程退出时静态变量自动析构，无需手动 `delete`，无内存泄漏
- C++11 标准保证局部静态变量初始化的线程安全
- 构造时自动输出日志 `"配置文件路径：..."`，便于诊断

### 3.2 加载流程

```
启动
  │
  ├─ InitFile::instance()         ← 首次调用，构造函数运行
  │    └ configFilePath = applicationDirPath() + "/config.json"
  │
  ├─ loadConfig()
  │    ├─ ensureConfigDirExists()
  │    ├─ isConfigExists() 否 → createDefaultConfig() → saveConfig()
  │    ├─ QFile 读盘 → QByteArray
  │    ├─ QJsonDocument::fromJson → configData
  │    ├─ validateConfig() 失败 → configData = getDefaultConfig() + saveConfig()
  │    └ emit configLoaded()
  │
  └─ 后续所有 getXxx() 仅读内存 configData，无磁盘 IO
```

### 3.3 验证与兜底

`validateConfig()` 检查 5 个必要节（database / camera / recognition / parking / system）和 database 内的必填字段。验证失败时用 `getDefaultConfig()` 覆盖并写盘，保证程序不会因配置崩溃。

**兼容性兜底**（`getFreeMinutes`）：

```cpp
int InitFile::getFreeMinutes() const {
    return configData["parking"].toObject()["freeMinutes"].toInt(15);
    //                                                              ^^ 默认值
}
```

旧版 `config.json` 缺少 `freeMinutes` 字段时自动回退为 15 分钟，避免崩溃。

### 3.4 配置变更信号

```cpp
// initfile.h
signals:
    void configLoaded();
    void configSaved();
    void configError(const QString &errorMsg);
    void parkingConfigChanged(const QString &name, double price, int capacity, int freeMinutes);
```

`setParkingConfig()` 在更新内存数据后发射 `parkingConfigChanged`，`main.cpp` 上行驶信号槽联动数据库：

```cpp
// main.cpp
QObject::connect(&initFile, &InitFile::parkingConfigChanged,
    [&dbManager](const QString &name, double price, int capacity, int freeMinutes) {
        if (dbManager.isConnected()) {
            dbManager.updateParkingConfig(name, price, capacity);
        }
    });
```

### 3.5 与数据库的初始化衔接

`main.cpp` 启动时序：

```cpp
InitFile &initFile = InitFile::instance();       // ① 构造单例
loadConfigWithRetry(&initFile);                    // ② 加载/创建配置
dbManager.connectDatabase(initFile.getDbHost(),   // ③ 连数据库
                          initFile.getDbPort(), ...);
MySQLInit mysqlInit(&dbManager, &initFile);        // ④ 注入单例
mysqlInit.initAll();                               // ⑤ 建表 + 初始数据
```

`MySQLInit::initAll()` 内部读取：

```cpp
initParkingData(initFile->getParkingName(),
                initFile->getParkingCapacity(),
                initFile->getParkingPrice());
```

数据库 PARKING 表初始数据与 `config.json` 完全一致。

## 4. 核心问题与解决方案

### 4.1 问题一：重复磁盘 IO

**现象**：每辆车出入库，日志出现多组 `"配置文件加载成功" + "配置文件保存成功"`。

**根因**（阶段一）：
- `checkIn()` / `checkOut()` / `calculateFee()` / `onUpdateParkingCount()` 各处 `new InitFile` + `loadConfig()`
- `loadConfig()` 内 `validateConfig()` 若失败会自动调用 `saveConfig()`
- 多次读盘 + 无意义写盘，性能差且日志嘈杂

**解决方法**：
1. 改为 Meyers 单例，启动时一次性 `loadConfig()` 读盘
2. `calculateFee` 改为参数传入（`hourlyRate` / `freeMinutes`），不再读配置
3. 删除 `DatabaseManager::getParkingNameFromConfig()` 辅助方法
4. `MainWindow` 读一次配置后通过参数向下传递

**效果**：一次出库从 2 次磁盘 IO（`loadConfig` × 2）降至 0 次（仅内存读取）。

### 4.2 问题二：数据库 PARKING 表写入硬编码值

**现象**：用户在配置窗口填写停车场名/容量/费率，config.json 正确保存，但数据库 PARKING 表始终是 `"默认停车场" / 100 / 5.0`。

**根因**（`MySQLInit::initAll()`）：

```cpp
// 旧代码
initParkingData("默认停车场", 100, 5.0);   // 硬编码
```

`MySQLInit` 看不到 `InitFile`，只能写死默认值。

**解决方法**：`MySQLInit` 构造函数增加 `InitFile*` 依赖，`initAll()` 内改用：

```cpp
initParkingData(initFile->getParkingName(),
                initFile->getParkingCapacity(),
                initFile->getParkingPrice());
```

**效果**：数据库初始数据与配置文件一致。

### 4.3 问题三：出入库后车位计数不更新

**现象**：车辆入库后 CAR 表插入成功，但 `PARKING.P_now_count` 始终为 0。

**根因**（`DatabaseManager::checkIn()` / `checkOut()`）：

```cpp
// 旧代码 — 只有 INSERT/UPDATE CAR
insertQuery.prepare("INSERT INTO CAR (license_plate, check_in_time) VALUES (...)");
// 没有 UPDATE PARKING SET P_now_count = P_now_count ± 1
```

两张表（CAR 流水表 / PARKING 汇总表）的写入被遗忘了一半。

**解决方法**：
1. `checkIn` / `checkOut` 签名增加 `const QString &parkingName` 参数
2. 事务内增加：

```cpp
// checkIn 内
UPDATE PARKING SET P_now_count = P_now_count + 1 WHERE P_name = :name

// checkOut 内
UPDATE PARKING SET P_now_count =
    CASE WHEN P_now_count > 0 THEN P_now_count - 1 ELSE 0 END
WHERE P_name = :name
```

3. 计数更新失败则整个事务回滚，保证 CAR/PARKING 数据一致

**效果**：每次出入库后 `P_now_count` 实时更新，UI 车位图表正确。

### 4.4 问题四：车牌含分隔符校验失败

**现象**：用户输入 `"京A·12345"`（中间含中点分隔符 U+00B7），`isValidLicensePlate()` 返回 false。

**根因**：

```cpp
QRegularExpression regex("^[京津沪渝][A-Z][A-Z0-9]{5,6}$");
//                                        ^^^^^^^^^^^^
//                                        不含中点，也不含空格/连字符
```

**解决方法**：增加静态归一化方法：

```cpp
QString Car::normalizePlate(const QString &input) {
    QString plate = input.trimmed();
    plate.remove(QStringLiteral("·"));   // 中点分隔符
    plate.remove(QStringLiteral(" "));   // 空格
    plate.remove(QStringLiteral("-"));   // 连字符
    plate = plate.toUpper();             // 统一大写
    return plate;
}
```

UI 层在校验前先调用 `normalizePlate()`，保证入库车牌为标准格式：

```cpp
QString plate = Car::normalizePlate(ui->entryPlateInput->text());
if (!Car::isValidLicensePlate(plate)) { ... }
```

**效果**：`京A·12345` → `京A12345` 入库成功，数据库存储统一格式。

## 6. 使用规范

### 6.1 读取配置

```cpp
// ✅ 正确：通过单例访问，仅读内存
QString name = InitFile::instance().getParkingName();
double price = InitFile::instance().getParkingPrice();

// ❌ 错误：new 栈对象（破坏单例唯一性）
InitFile cfg;
cfg.loadConfig();   // 多余读盘
```

### 6.2 写入配置

```cpp
// ✅ 正确：setter 内部自动 emit + saveConfig()
InitFile::instance().setParkingConfig(name, price, capacity, freeMinutes);

// ❌ 错误：外部直接改 configData（绕过封装）
// configData 是 private 成员，外部无法访问
```

### 6.3 向下传参

```cpp
// ✅ 正确：UI 层读一次，参数向下传
void MainWindow::onEntrySearchButton() {
    QString parkingName = getParkingNameFromConfig();  // UI 层职责
    m_db->checkIn(plate, parkingName);                 // 参数下传
}

// ❌ 错误：数据库层自己读配置（耦合）
bool DatabaseManager::checkIn(const QString &plate) {
    InitFile cfg;              // 数据库层不该知道配置
    cfg.loadConfig();
    QString name = cfg.getParkingName();
    ...
}
```

### 6.4 线程安全

Meyers 单例本身是线程安全的（C++11 保证），但 `loadConfig()` 与 `saveConfig()` 若在多线程交叉运行仍有风险。当前项目为单线程 Qt 事件循环，不构成问题；若未来引入多线程配置热更新，需增加 `QReadWriteLock` 保护。
