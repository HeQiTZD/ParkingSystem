# 开发问题文档

## 问题 #1：CameraManagement 页面摄像头画面颜色异常

| 项目 | 内容 |
|------|------|
| **日期** | 2026-07-20 |
| **严重程度** | 中 |
| **影响范围** | CameraManagement 页面所有摄像头窗口的视频画面 |
| **发现方式** | 用户反馈 |
| **修复状态** | ✅ 已修复 |

---

### 1. 问题描述

摄像头管理页面 (`CameraManagement`) 中的摄像头视频画面颜色不是彩色，呈现红蓝通道互换的偏色画面。相同摄像头在主页 (`MainWindow`) 中显示正常。

---

### 2. 根因分析

**根本原因：`cv::Mat` 浅复制 + 原地 `cvtColor` + 多路订阅同一信号，导致共享像素缓冲区被串行修改。**

#### 2.1 数据流

```
CameraThread::run()                          ← 工作线程
  │
  └─ emit newFrameCaptured(frame.clone())   ← 深度克隆一份 BGR 帧 (Data_BGR)
       │
       │  cv::Mat 的复制构造函数是浅复制——只复制 Mat 头部（共享像素数据）
       │  Qt QueuedConnection 对信号参数做复制存储时，产生多份指向同一缓冲区的 Mat
       │
       ├─ QueuedConnection ──► MainWindow::updateFrame     ← 主线程，先执行
       │                          cvtColor(frame, frame, BGR2RGB)  ← 原地修改共享缓冲区
       │                          Data_BGR → Data_RGB ✓ 正确显示
       │
       └─ QueuedConnection ──► CameraWindow::onFrame       ← 主线程，后执行
                                  cvtColor(frame, frame, BGR2RGB)  ← 再次原地修改
                                  Data_RGB → Data_BGR ✗ BGR 数据被当作 RGB888 渲染
```

#### 2.2 关键机制

1. **`cv::Mat` 浅复制**：拷贝构造只复制 Mat 元数据头（data、step、size等），不复制像素内容。多个 Mat 对象引用同一块像素缓冲区，通过引用计数管理生命周期。

2. **原地 `cvtColor`**：`cvtColor(frame, frame, COLOR_BGR2RGB)` 中 src 和 dst 是同一个 Mat 对象，直接在共享缓冲区上做通道交换（B↔R）。

3. **多路订阅同一信号**：设计文档明确指出 `newFrameCaptured` 被主页 `updateFrame` 和摄像头页 `CameraWindow::onFrame` 同时订阅。两个槽函数在主线程中顺序执行，第一个执行后已将共享缓冲区从 BGR 改为 RGB，第二个拿到的是已被修改的数据，再次执行 BGR→RGB 等于交换回 BGR。

#### 2.3 关联文件

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/camera/camerathread.cpp` | 129 | `emit newFrameCaptured(frame.clone())` — 数据源头 |
| `UI/CameraManagement/camerawindow.cpp` | 80 | 原地 `cvtColor(frame, frame, COLOR_BGR2RGB)` |
| `UI/MainWindow/mainwindow.cpp` | 283 | 原地 `cvtColor(frame, frame, COLOR_BGR2RGB)` |
| `UI/CameraManagement/camerawindow.cpp` | 51 | `connect(m_thread, &CameraThread::newFrameCaptured, ...)` — 第二个订阅者 |
| `UI/MainWindow/mainwindow.cpp` | 160 | `connect(m_cameraThread, &CameraThread::newFrameCaptured, ...)` — 第一个订阅者 |

---

### 3. 影响评估

#### 3.1 已确认影响
- CameraManagement 页面摄像头画面红蓝通道互换，视觉上颜色异常
- 主页和摄像头页面**同时存在**时触发，两者独立运行时均正常

#### 3.2 不影响项
- **车牌识别不受影响**：识别数据通路是 `CameraThread → FrameQueue → RecognizeThread`，每个环节均使用 `clone()` 深拷贝，与显示通路完全隔离
- 摄像头捕获、帧率统计、状态显示等功能不受影响

---

### 4. 修复方案

将两个槽函数中的原地 `cvtColor` 改为使用独立的 `cv::Mat` 作为转换目标：

```cpp
// 修复前
cv::cvtColor(frame, frame, cv::COLOR_BGR2RGB);
QImage img(frame.data, ...);

// 修复后
cv::Mat rgbFrame;
cv::cvtColor(frame, rgbFrame, cv::COLOR_BGR2RGB);
QImage img(rgbFrame.data, ...);
```

#### 4.1 修改文件

| 文件 | 修改内容 |
|------|----------|
| `UI/CameraManagement/camerawindow.cpp` | `cvtColor(frame, frame, ...)` → `cvtColor(frame, rgbFrame, ...)`，QImage 使用 `rgbFrame` |
| `UI/MainWindow/mainwindow.cpp` | `cvtColor(frame, frame, ...)` → `cvtColor(frame, rgbFrame, ...)`，QImage 使用 `rgbFrame` |

---

### 5. 经验教训

1. **跨信号槽传递大型数据时需警惕浅复制**：`cv::Mat` 的拷贝语义是共享而非独立，多接收者场景下任一接收者对数据的修改都会影响其他接收者。

2. **避免 OpenCV 颜色空间转换的原地操作**：即使单接收者场景下原地转换可能可行，但使用独立目标 Mat 更安全、可读性更好，且能避免潜在的数据竞争。

3. **多路订阅时的数据隔离**：当同一数据源被多个消费者订阅时，每个消费者应在本地创建独立副本再处理，而非直接操作共享数据。
